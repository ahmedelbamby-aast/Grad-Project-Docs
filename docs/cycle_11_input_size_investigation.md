# Cycle 11 — Behavior Input Size 320 → 256 (Investigation)

**Last updated:** 2026-06-02

**Status:** **PHASE 3 COMPLETE — 11.A NOT ACCEPTED.** The 256 candidate was
built on production, failed the pre-benchmark parity gate, and was rolled back
to the accepted 320 Top-K baseline. See
[`docs/cycle_11_input_size_results.md`](cycle_11_input_size_results.md).

This document is the Cycle 11 Phase 1 design + risk analysis required before
any engine rebuild. It is the §B.5 "name the lever" output for the Cycle 11
candidate ranked #1 by the
`docs/cycle_9b_child_critical_path_remeasure_topk_results.md`
remeasurement.

## Why Cycle 11 now

Cycle 9b B.2.c (exact slice + Top-K) is the current accepted production
baseline (job `be4ba9ee`, 4.429 FPS, 17.0 min total wall, Step 2 frame
wall 540.4 s). The Top-K caveat is explicit: **average GPU utilization did
not improve** (`9.595 % → 9.3 %`). The remeasurement in
`docs/cycle_9b_child_critical_path_remeasure_topk_results.md` identified
two facts that demand a per-crop-compute attack:

1. **All four base behavior children now run at essentially the same
   per-execution cost** under the Top-K topology (~30 ms each at bs=32
   prod aggregate, ~16-19 ms at bs=17 probe). The dominant child gap
   shrank from +33 % to +15 %. The per-child optimization ceiling
   is **~4 % Step 2 wall**.
2. **Per-crop GPU compute is constant at ~0.94 ms/crop** across batch
   sizes. The lever is per-crop work, not per-call overhead.

Cycle 11 attacks per-crop work directly by halving the pixel budget
(320² = 102 400 → 256² = 65 536, a 36 % reduction). Per the
`docs/cycles_9_to_12_implementation_playbook.md` §4 projection, this
should drop per-child server compute from ~14 ms to ~9 ms at production
batch size, yielding **−10 % to −13 % Step 2 wall**.

The §E.6 multi-approach rule means Cycle 11 must be benchmarked against
the lower-risk fallback B.3.b/d (kernel-tactic tuning on the dominant
child), even though that fallback is bounded at ~4 % Step 2 wall.

## Named lever (§B.5)

**GPU child compute (across all 4 behavior children).** Pixel volume is
the input to convolution work; halving it reduces per-crop GPU work
roughly linearly because the dominant ops are 2D convs whose FLOPs scale
as `O(H × W × C_in × C_out × K²)`.

## Mechanism

| Step | What changes | Files touched |
|---|---|---|
| 1 | Re-export 4 base behavior engines at imgsz=256 | `backend/models/standing_sitting/weights/standing_sitting.engine`, `right_left/weights/right_left.engine`, `up_down/weights/up_down.engine`, `forward_backward/weights/forward_backward.engine` |
| 2 | Re-export `gaze_horizontal_slice_model` at imgsz=256 (so it consumes the 1344-anchor dense output of the new gaze_h base) | `right_left/weights/right_left_slice.engine` |
| 3 | Re-export 4 Top-K adapters at imgsz=256 (input shape becomes `[N, C, 1344]` instead of `[N, C, 2100]`) | `standing_sitting_topk.engine`, `right_left_slice_topk.engine`, `up_down_topk.engine`, `forward_backward_topk.engine` |
| 4 | Update Triton configs `dims: [ 3, 320, 320 ] → [ 3, 256, 256 ]` for all ensembles + base children + slice adapters | All `config.pbtxt` files listed below |
| 5 | Update env: `TRITON_CROP_BEHAVIOR_INPUT_SIZE=256` | `backend/.env` (prod) |
| 6 | Reload Triton, restart Celery workers | runtime |

### Anchor count math

YOLO grid anchors = `(H/8)² + (H/16)² + (H/32)²` for square input:

| imgsz | Anchors | Δ vs 320 |
|---|---:|---:|
| 320 (current) | 2 100 | — |
| **256 (Cycle 11)** | **1 344** | **−36 %** |
| 192 (stretch, NOT planned) | 756 | −64 % |

The Top-K value (`TRITON_BEHAVIOR_TOP_K_VALUE=100`) is well under the
new 1 344-anchor count, so no Top-K value change is required.

### Files that hard-code `dims: [ 3, 320, 320 ]` and need updating

```
backend/models/triton_repository_cuda12/behavior_ensemble/config.pbtxt
backend/models/triton_repository_cuda12/behavior_ensemble_gaze2/config.pbtxt
backend/models/triton_repository_cuda12/behavior_ensemble_gaze_slice/config.pbtxt
backend/models/triton_repository_cuda12/behavior_ensemble_gaze_slice_topk/config.pbtxt
backend/models/triton_repository_cuda12/gaze_horizontal_gaze2_model/config.pbtxt
backend/models/triton_repository_cuda12/gaze_horizontal_slice_adapter/config.pbtxt
```

Plus per-base-child configs are regenerated by
`build_tensorrt_engines.py` automatically. The slice/topk-adapter
configs are likewise regenerated by that script — they encode the
anchor count from `imgsz`.

The 6 ensemble configs above hard-code the image dims, so a Cycle 11
build needs either (a) a manual edit pass or (b) a helper that rewrites
the `images` input dims of each ensemble config in place. **Approach (b)
is preferred** — it survives future input-size changes without manual
work and is rollback-safe (the rewrite is idempotent).

## Implementation order

Following the constitution's "no code without a plan":

### Phase 2 — build (next commit)

1. Add a helper `tools/prod/prod_set_behavior_input_size.sh` that:
   - Validates the requested size (≥ 32, % 32 == 0, ≠ current).
   - Backs up the 6 ensemble `config.pbtxt` files to `*.bak.<size>`.
   - Rewrites their `dims: [ 3, <old>, <old> ]` to `dims: [ 3, <new>, <new> ]`.
   - Calls `prod_enable_roi_crop_behavior.sh --input-size <new>` (base
     engines).
   - Calls `prod_enable_behavior_topk.sh --input-size <new> --topk 100`
     (slice + Top-K adapters at the matching anchor count).
   - Restarts Triton and Celery.
2. Unit test the helper's config rewriter (pure-Python rewrite of a
   sample config string) under
   `backend/tests/unit/pipeline/test_behavior_input_size_helper.py`.

### Phase 3 — accuracy parity gate (mandatory before benchmark)

Reuse the existing decode pipeline. Run **before** the prod benchmark.

```bash
# 200 random crops from combined.mp4 fed through both 320 engines and
# 256 engines; compare top-1 class, centroid drift, and class confidence
# delta on the SAME crops.

ssh prod-grad 'cd /home/bamby/grad_project && \
  TRITON_CROP_BEHAVIOR_INPUT_SIZE=256 \
  backend/.venv/bin/python tools/prod/prod_behavior_input_size_parity.py \
    --crops 200 \
    --source combined.mp4 \
    --host 127.0.0.1:39101 \
    --json backend/logs/parity_input_size_256_$(date -u +%Y%m%dT%H%M%S).json'
```

### Acceptance gates (all three must pass)

| Gate | Threshold | Why |
|---|---|---|
| Top-1 class agreement (per behavior model) | ≥ 99.5 % | Catches gross classification drift from smaller receptive field |
| Bounding-box centroid drift (normalized) | ≤ 0.5 px at 256 scale | Catches subtle localization drift |
| Per-class confidence delta (absolute) | ≤ 5 % per class | Catches confidence collapse |

Only if all three pass: run the full `combined.mp4` benchmark.

### Phase 4 — production benchmark gates

| Gate | Threshold | Source |
|---|---|---|
| Decoded parity vs Top-K baseline (be4ba9ee) | per-class bbox counts ±1 % (looser than Cycle 9b's ±0.05 % because the engines actually differ) | DB |
| Frame rows count | unchanged | DB |
| RTMPose `pose_record_count` | ±0.1 % | inference_audit |
| StudentTrack count | unchanged | DB |
| Step 2 wall reduction | ≥ 10 % vs 540.399 s (Top-K baseline) | bench summary |
| Overall FPS | ≥ 4.429 (Top-K baseline) | bench summary |
| Behavior RTT mean | ≤ 84.865 ms (Top-K baseline) | inference_audit |
| Average GPU utilization | > 9.3 % (Top-K baseline) — primary caveat target | GPU CSV |

The GPU-util gate is added explicitly because Cycle 9b's caveat was that
GPU util did not improve. Cycle 11 attacks per-crop compute, so if it
works, GPU util should rise (smaller per-call compute → more calls per
second → higher steady-state util).

## Rollback

One env flip + Triton restart:

```bash
ssh prod-grad 'cd /home/bamby/grad_project && \
  bash tools/prod/prod_set_behavior_input_size.sh --input-size 320 && \
  bash tools/prod/prod_start_triton.sh && \
  bash tools/prod/prod_start_celery_workers.sh'
```

The 320 engines are never deleted — they live in
`backend/models/.../*.engine.bak.320` plus the original Triton plan
location (the rebuild process overwrites `model.plan` with the new
`.engine`, so we keep the prior plan in a versioned subdirectory).

## Risk

| Risk | Likelihood | Mitigation |
|---|---|---|
| Accuracy parity gate fails (smaller receptive field misclassifies edge cases) | medium | Mandatory pre-benchmark parity probe; reject if any of 3 gates miss |
| TRT engine build fails on prod (workspace too small, kernel selection) | low | The existing `prod-rebuild-tensorrt-engines.sh` has handled this surface for the 320 build; same machine + driver |
| Ensemble config edit corrupts the file | low | Pure-Python rewrite covered by unit test; backup file on every edit |
| Top-K adapter shape mismatch at the 1 344-anchor input | low | `build_tensorrt_engines.py` derives anchors from imgsz; matching profile already implemented at line 213-218 |
| GPU util still doesn't improve (per-crop work isn't the binding constraint after all) | medium | The benchmark itself is the test; if GPU util stays flat AND Step 2 wall doesn't drop, REJECTED with documented evidence |
| Cycle 11 conflicts with future Cycle 12+ work | low | Cycle 12 (parallel render + COPY FROM) is orthogonal to behavior engine size |

## Multi-approach matrix (per §E.6)

Per §E.6 of the TODO file, BOTH top-ranked candidates must be prod-
benchmarked before one is selected. The matrix is:

| ID | Approach | Lever | Projected Step 2 wall Δ | Engineering effort | Risk |
|---|---|---|---|---|---|
| **11.A** | Cycle 11 input 320 → 256 (this doc) | per-crop GPU compute (all 4) | **−10 % to −13 %** | ~2 person-days | medium (accuracy parity required) |
| **11.B** | B.3.b batch-profile + B.3.d kernel-tactic tuning on `gaze_horizontal_model` only | per-call GPU compute (dominant only) | ≤ −4 % (bounded by 2.47 ms gap) | ~1.5 person-days | low (no precision change) |

Selection rule:

- If 11.A passes its accuracy parity gate AND delivers ≥ 10 % Step 2
  wall, **11.A ships**.
- If 11.A fails accuracy parity OR fails the Step 2 gate, fall back to
  11.B (lower ceiling but lower risk).
- Document both runs in `docs/cycle_11_input_size_results.md` regardless
  of which ships.

Env-flag selection (rollback is a single env flip in either direction):

```env
# Defaults — current accepted Top-K baseline
TRITON_CROP_BEHAVIOR_INPUT_SIZE=320          # 11.A flag: set to 256
GAZE_HORIZONTAL_ENGINE_VARIANT=fp16          # 11.B flag: set to e.g. fp16_kernel_tuned
```

`GAZE_HORIZONTAL_ENGINE_VARIANT` is the per-child variant flag pre-
declared by §B.3 of the TODO. Setting it to `fp16_kernel_tuned`
selects the B.3.b/d rebuild of `gaze_horizontal_model` with explicit
TRT builder tactic hints + batch-profile alignment.
